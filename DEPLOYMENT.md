# Deployment Guide

This project is a monorepo with two deployable units:

| Unit | Path | Host | Why |
|------|------|------|-----|
| **Frontend** (Vue 3 + Vite) | `frontend/` | **Vercel** | Static SPA build, global CDN |
| **Backend** (FastAPI + WebSockets) | `backend/` | **Render** | Needs persistent processes + WebSockets (not serverless-friendly) |
| **Database** (Postgres) | — | **Neon** | Already provisioned |

> **Why not Vercel for the backend?** The CRM Kanban uses WebSockets for live updates, and SQLAlchemy holds a connection pool. Vercel's serverless functions are stateless with short timeouts and no persistent socket support, so the API runs on Render instead.

---

## 1. Database (Neon) — already done

Two roles are in use (see `backend/scripts/provision_roles.sql`):

- `app_user` (**NOBYPASSRLS**) — runtime connections; Row-Level Security is enforced.
- owner role — migrations only (DDL).

Migrations `0001`–`0005` are already applied to the production `neondb`. The Render build re-runs `alembic upgrade head`, which is idempotent (no-ops if already current).

Connection string format (asyncpg): use `?ssl=require`, **not** `sslmode=require`.

```
postgresql+asyncpg://USER:PASSWORD@HOST/neondb?ssl=require
```

---

## 2. Backend → Render

`backend/render.yaml` is a Blueprint. In the Render dashboard: **New → Blueprint**, point it at this repo. It picks up `rootDir: backend` automatically.

### Environment variables to set (marked `sync: false`)

| Var | Value |
|-----|-------|
| `DATABASE_URL` | Neon **app_user** URL (`...?ssl=require`) |
| `MIGRATION_DATABASE_URL` | Neon **owner** URL (`...?ssl=require`) |
| `GOOGLE_CLIENT_ID` | From Google Cloud OAuth client |
| `GOOGLE_CLIENT_SECRET` | From Google Cloud OAuth client |
| `GOOGLE_REDIRECT_URI` | `https://<service>.onrender.com/auth/google/callback` |
| `FRONTEND_URL` | `https://<your-app>.vercel.app` |
| `CORS_ORIGINS` | `https://<your-app>.vercel.app` (add `,http://localhost:5173` for local dev) |

`JWT_SECRET` and `SESSION_SECRET` are auto-generated by Render (`generateValue: true`). `PYTHON_VERSION` is pinned to 3.12.7.

Health check: `GET /health` → `{"status": "ok"}`.

> **Free tier note:** Render free web services spin down after ~15 min idle; the first request after sleep takes ~30–60s to wake.

## 3. Frontend → Vercel

In the Vercel dashboard: **Add New → Project**, import this repo, then **set Root Directory to `frontend`**. Vercel reads `frontend/vercel.json` (build command, SPA rewrites).

### Environment variables

| Var | Value |
|-----|-------|
| `VITE_API_URL` | `https://<service>.onrender.com` |

The WebSocket URL is derived automatically (`https`→`wss`). Only set `VITE_WS_URL` if your WS host differs from the API host.

> Set `VITE_API_URL` for **Production** (and Preview, if you want preview deploys to hit the same API).

---

## 4. Order of operations

1. Deploy the **backend** first (Render) → note its `https://<service>.onrender.com` URL.
2. Deploy the **frontend** (Vercel) with `VITE_API_URL` = that URL → note the `https://<app>.vercel.app` URL.
3. Go back to Render and set `FRONTEND_URL` + `CORS_ORIGINS` to the Vercel URL → redeploy (or just save; Render restarts).
4. In **Google Cloud Console** → OAuth client → add the authorized redirect URI `https://<service>.onrender.com/auth/google/callback` and the JS origin `https://<app>.vercel.app`.

---

## 5. Smoke test after deploy

1. Open the Vercel URL → sign up → confirm the 5 default pipeline stages appear.
2. Create a lead → drag it between Kanban columns.
3. Open the app in a second tab → drag a card in tab 1 → confirm tab 2 updates live (WebSocket).
4. Open a lead → add a note + a task.

If step 3 fails, check the browser console for a blocked `wss://` connection (usually a `CORS_ORIGINS` / mixed-content issue) and the Render logs.
